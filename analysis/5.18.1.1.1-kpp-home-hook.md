# Kindle 5.18.1.1.1 Home-card hook analysis

## Result

On this firmware, the visible Home screen is rendered by `/app/bin/KPPMainAppV2`, not by the Java `home.jar`. The KPM tweak target should remain the built-in target `pillow`: `pillowd` is the daemon executable on this image, while `pillow` is the framework target identity whose descendants receive the preload environment.

The preferred card injection point is:

```text
KPPHomeApp::CardAggregator::getAggregateCards()
RVA 0x011265b0 in KPPMainAppV2
```

Call the original, convert each KPM manifest entry into a native `CardModel`, append the resulting `shared_ptr<CardModel>` to the returned vector, and let KPP's existing sequencer validate, order, and serialize the cards.

KPP exports the conversion helper needed for this:

```text
KPPHomeApp::DataProviderUtil::convertJSONToCardModel(utils::json::JsonContainer)
RVA 0x01139a50
```

The conversion function accepts a JSON object with the native model fields `CardId`, `WidgetId`, `CardData`, `CardTemplateId`, `CardStatus`, `CardPosition`, and `CardPriority`. Its disassembly shows that missing position and priority are initialized to `-1`, so KPP can assign placement.

The launch interception point is:

```text
KPPNavigation::NavigationModule::navigateTo(
    std::string,
    utils::json::JsonContainer)
RVA 0x011e2d34
```

Synthetic cards should carry a reserved URI such as `kpm-app://com.example.package/app`. The navigation hook recognizes that scheme, calls `ksubstrate_home_apps_launch(id)`, and delegates every other URI unchanged to the original function.

## Why this seam

The Hermes bundle calls `NativeModules.HomeUIDataProvider.GetCards()` to populate the Home card list and `NativeModules.LibraryDataProvider.GetLibraryCard()` for the safe-home fallback. Native symbols show the flow below:

```text
HomeUIDataProvider::getCards()
  -> HomeOrchestrator::getOrderedCardsForHomeUI()
    -> CardAggregator::getAggregateCards()
      -> each IDataProvider::getCardsFromProvider()
    -> CardSequencer::sortCards(...)
```

Injecting at the aggregator boundary is earlier than final serialization but later than the individual Amazon data providers. It therefore avoids all of the following:

- creating placeholder files in `/mnt/us/documents`;
- fabricating a `ksdk::library::LibraryItem` and entering the content database;
- patching the Hermes bytecode;
- depending on the library provider's asynchronous refresh implementation;
- replacing the entire Home response.

`LibraryDataProvider::getCardsFromProvider()` at RVA `0x0113618c` is a useful fallback if the product decision is to place KPM apps specifically inside the existing "From Your Library" card. It is less attractive for the first implementation because its output and refresh behavior are coupled to the library card's nested schema.

## Card shape and icon handling

`convertJSONToCardModel` only constructs KPP's outer native model. The selected `CardTemplateId` determines the required contents of `CardData`; KPP exports validators and trimmers for templates 2, 5, 9, 12, 13, 14, 17, 18, 20, 21, and 49. Before enabling a production card, capture one existing, simple card after aggregation and use its trimmed JSON as the template contract. Replace only the stable fields (ID, title, image URI, and navigation URI).

The KPM icon can remain inside the installed package. The synthetic card should reference the resolved absolute icon path using the image URI form accepted by the chosen KPP template. No document-directory artifact is required.

## Targeting and ABI constraints

- Keep `targets: ["pillow"]` in the tweak manifest.
- Confirm on-device that the preload environment inherited from the `pillow` framework target reaches `KPPMainAppV2`; if the launcher sanitizes it, add KPP as an explicit executable target rather than renaming the built-in target to `pillowd`.
- These functions use the ARM hard-float C++11 ABI and Thumb-2 code. Returned `vector` and `shared_ptr` values use hidden structure-return parameters.
- Implement the KPP-facing bridge in C++ compiled against the matching libstdc++ ABI. Keep the Rust registry and launch API behind a small C interface.
- Resolve exported mangled symbols with `dlsym` where possible, but retain the checked RVAs and prologues in `symbols.5.18.1.1.1-kpp-home.yaml` as the firmware-specific fallback.
- Refuse to install hooks when the prologue signature does not match.

## Evidence pulled from the device

The read-only device pull is under `analysis/pulled/5.18.1.1.1/rootfs` (ignored by Git). It includes `KPPMainAppV2`, the Hermes bundle, `pillowd`, `appmgrd`, relevant shared libraries, Java Home artifacts, and `appreg.db`.

The X11 tree identified the full-screen viewable window as `com.lab126.KPPMainApp`. `appreg.db` identified that application as the default application association, while `com.lab126.booklet.home` remains a Java booklet/detail provider. The Hermes bundle is HBC version 84 and directly references `HomeUIDataProvider.GetCards`, `LibraryDataProvider.GetLibraryCard`, and `NavigationModule.navigateTo`.

This analysis is static and read-only. No hook was installed or exercised on the Kindle, so the remaining hardware proof is a logging-only hook followed by one synthetic card and launch test.
